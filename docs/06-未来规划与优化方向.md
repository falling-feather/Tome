## 0. 评估方法与全局结论

走查范围：[backend/app](../backend/app)（约 9 个 service + 5 个 router）、[frontend/src](../frontend/src)（9 个 page + 6 个 component）、`docker-compose.yml`、`scripts/*`。

**总体判断：**

- 后端架构分层清晰、模块职责合理，叙事管线（事件池 / Prompt 装配 / 记忆 / 审计）是项目最大的工程亮点；
- **生产就绪度 ≈ 70%**：核心功能完备，但在密钥安全、并发安全、迁移管理、自动化测试、前端错误边界上仍有明显缺口；
- 短期最优投入方向是 **P0 安全/数据迁移** 与 **P1 测试与可观测性**，这两块直接影响后续所有迭代的可靠性。

## 1. 安全与合规（最高优先级）

### P0-1 默认管理员密码硬编码（暂时不考虑（最高否定优先级））

- **位置**：[backend/app/main.py](../backend/app/main.py) `seed_admin()` 中 `password_hash=hash_password("mic820323")`。
- **风险**：任何 fork / 部署都会保留同一弱口令，等同后门。
- **建议**：
  1. 启动时若环境变量 `ADMIN_INITIAL_PASSWORD` 缺失则**生成随机密码并打印到日志一次**；
  2. 首次登录强制改密；
  3. 仓库根目录 `apikey` 文件应纳入 `.gitignore`（当前根有 `apikey` 文件存在，需要确认是否含真实密钥）。

### P0-2 SECRET_KEY 默认值是字面量

- **位置**：`config.py` 默认 `"change-this-in-production"`。
- **建议**：启动时若仍为默认值且非 DEBUG，**直接 raise** 中止启动，强制用户配置。

### P0-3 用户 API Key 明文存储（暂时不考虑（最高否定优先级））

- **位置**：`models.UserApiKey.api_key`，普通字符串列。
- **风险**：DB 文件泄露 = 全部用户密钥泄露。
- **建议**：
  - 用 `cryptography.fernet` 在 `auth.py` 同级新增 `secrets.py`，写入/读取时透明加解密；
  - 主密钥来自 `SECRET_KEY` 派生，或独立 `KEY_ENCRYPTION_KEY` 环境变量。

### P1-4 CORS `allow_origins=["*"]` + `allow_credentials=True`

- **位置**：`main.py`。
- **风险**：浏览器规范上两者并存浏览器会忽略 credentials；同时给生产留下 CSRF 隐患。
- **建议**：用 `CORS_ORIGINS` 环境变量白名单替换。

### P1-5 关键路由日志写入失败不会回滚

- 多处路由 `db.add(log); await db.commit()` 与业务写入混在同一事务后单独 commit，若日志失败可能影响响应。
- **建议**：抽出 `audit_log()` 帮助函数 + 独立 try。

### P2-6 输入校验粒度

- `ActionInput.content` 限制 1–2000 字符，但未做基础敏感词 / 注入字符过滤；自编故事 50000 字符上限较大，单用户可较快耗费 LLM 配额。
- **建议**：增加按用户的"日 Token 配额"指标 + 软上限。

## 2. 数据与持久化

### P0-7 缺少数据库迁移

- 当前用 `Base.metadata.create_all` 在启动时建表；新增字段会**静默不生效**。
- **建议**：引入 **Alembic**，把现有 schema 作为 `001_initial`；CI 内 `alembic upgrade --sql` 校验。

### P1-8 SQLite 并发写入风险

- 默认 `aiosqlite`，记忆压缩、SSE、活动日志均为并发写；高并发下会出现 `database is locked`。
- **建议**：
  - 文档中明确"生产请使用 PostgreSQL"（已部分体现）；
  - 后端 `engine` 增加 `pool_pre_ping`、PG 时启用 `pool_size`；
  - 短事务化：将 SSE 中"逐 chunk yield"与"最终 commit"拆分，避免长事务持有连接。

### P1-9 `events_triggered` 列表无界裁剪到 50 后仍 JSON 序列化

- 多用户长会话下 `state` JSON 体积单调增长，记忆压缩并不裁剪 `events_triggered` 与 `event_cooldowns` 之外的字段。
- **建议**：state 中"统计型"字段（如 `npcs_mentioned` 累计、`affection_scores`）独立成表。

### P2-10 `game_events` 表已建但完全未使用

- 事件以 `EVENTS` Python 常量管理，admin 无法在线编辑。
- **建议**：迁移到 DB（与世界书一致），后台提供 CRUD；保留 Python 常量作为 `seed_if_empty()`。

### P2-11 `messages.metadata_` 列名映射易踩坑

- ORM 字段 `metadata_` ↔ DB 列 `metadata`，新成员易写错；建议统一加 `# noqa` 注释和 schema 注释。

## 3. 叙事管线与 LLM 质量

### P1-12 PromptAssembler 与 system prompt 在每回合全量拼接

- 每个动作都重新查 `world_entries` + `prompt_templates` + `memory_entries`，三次 DB 查询。
- **建议**：以 `(scenario, chapter)` 为 key，给装配结果加 **进程内 LRU 缓存**（5 分钟过期），世界书 / 模板写操作主动 invalidate。

### P1-13 WorldBookService 仅基于关键词命中

- 缺少向量相似度，对模糊语义（"我去找会算命的"无法命中"占卜师"）召回率低。
- **建议**：引入轻量嵌入（`bge-small-zh` ≤ 100MB 或调用 SiliconFlow embedding 接口） + 内存向量索引（Chroma/SQLite-FAISS），关键词作为 fallback。

### P1-14 多智能体审计是"全部通过 LLM 重写"还是"规则 + 选择性重写"

- 当前 `AuditAgent` 触发后会调用 `llm_complete` 重写整段，这增加约 1× LLM 成本。
- **建议**：
  - 引入 `audit_severity_threshold` 环境变量；
  - 仅当 `total_issues ≥ N` 时才重写；
  - 给 `agents.py` 中硬编码的 `CLASS_RESTRICTIONS` / `SCENARIO_RESTRICTIONS` 等改为 DB 驱动，admin 可在线维护。

### P1-15 SSE 期间状态推断与审计在"全部生成完毕后"才开始

- 用户先看到 raw 文本，等 1–3s 才看到段落整理 / 状态变更。
- **建议**：
  - 段落标签解析可前端实时进行（已实现）；
  - 服务端审计支持 **"轻量规则同步检查 + 重量级 LLM 后置异步"**，前端先用 raw 渲染，收到 `audit` 事件后做差异 patch。

### P2-16 `post_processor` 正则集合过于贪婪

- "(?:获得|得到|捡起|拿到|收获|拾取|入手)(?:了)?\\W?(.{2,10})" 在叙事中容易把整段文字误抽为物品名。
- **建议**：要求 LLM 在 metadata 段落（`[META] items: ...`）中输出结构化变更，正则做兜底。

### P2-17 死亡判定未与 LLM 输出强耦合

- 状态从 combat 跳到 dead 仅靠事件 `effects` 或 `health<=0` 推断；LLM 自由叙事中描述"被一刀劈死"未被解析。
- **建议**：约定 `[STATE] dead=true` 元行；或在审计阶段增加"死亡叙事检测"。

## 4. 性能与并发

### P1-18 全局单例 Resilience 组件未做线程/异步隔离

- `CircuitBreaker.state` 是普通属性，多 worker 下不共享；`RateLimiter._user_requests` 也是进程内 dict。
- **建议**：
  - 单进程部署可接受；
  - 多 worker / 水平扩展时切换到 Redis（限流用 `INCR + EXPIRE`，断路器用 hash + ttl）。

### P2-19 `httpx.AsyncClient` 每次请求新建

- `_call_stream` 中 `async with httpx.AsyncClient(...)`，无连接复用。
- **建议**：服务级单例 client，超时仍可在 stream 调用上覆盖。

### P2-20 前端无防抖 / 无离线提示

- `GameChat` 在断网或 SSE 中断时只显示通用错误，未做指数重连。
- **建议**：引入 `EventSource`-like 重连封装，并提示"将自动重试 3 次"。

## 5. 可观测性与运维

### P1-21 日志格式非结构化

- `middleware.py` 用纯 logger.info；难以接入 Loki/ELK。
- **建议**：迁移到 `structlog` 或自定义 JSON formatter，输出 `request_id`/`user_id`/`session_id`。

### P1-22 缺少分布式 trace

- LLM 流 + 审计 + 记忆压缩链路较长。
- **建议**：OpenTelemetry SDK + OTLP 导出；先在本地 Jaeger 跑通。

### P2-23 健康端点信息不分级

- `/api/health` 公开返回 `llm_circuit` 状态，可能让攻击者窥探内部状态。
- **建议**：公共版只返回 `{status, version}`，详细指标放在 `/api/admin/health`（已有）。

### P2-24 Docker compose 单服务、无日志卷

- 容器重启会丢失 stdout 历史；缺少 PG 服务示例。
- **建议**：补充 `volumes: ./logs:/var/log/inkless`，附 `db: postgres:16` 可选 profile。

## 6. 工程化与 DX

### P1-25 完全没有自动化测试

- 仓库内未发现 `tests/` 目录或 CI 配置。
- **建议**：
  - 后端：pytest + pytest-asyncio + httpx ASGI client；
  - 重点覆盖 `game_engine.update_state`、`memory_service.check_and_compress`、`agents.AgentOrchestrator.process`、`post_processor.process`；
  - 前端：Vitest + RTL 至少覆盖 `parseSegmentsRealtime` 与 `client.submitAction` 的 SSE 解析。

### P1-26 类型注解不完整

- `services/llm_service.py` 多处 `kwargs` 透传无类型；前端 `api/client.ts` 大量 `Promise<any>`。
- **建议**：
  - 后端启用 `mypy --strict`（先 services 层）；
  - 前端把 `any` 收敛到 `types/api.ts`。

### P2-27 Lint / Format 未统一

- 缺 `ruff` / `prettier` / `eslint` 配置文件。
- **建议**：根目录加 `pyproject.toml` (ruff + black 替代品 ruff format) 与 `.prettierrc`。

### P2-28 README 与 main.py 版本号不一致

- README 写 v1.0，`FastAPI(title=..., version="0.1.0")` 仍是 0.1.0；`/api/health` 也返回 `"version":"0.1.0"`。
- **建议**：把版本号集中到 `config.py`（如 `APP_VERSION = "1.0.0"`）并在 `main.py`、`/api/health`、前端 footer 引用。

### P3-29 `scripts/deploy.ps1` 编码隐患

- 既有用户记忆显示 PS5.1 默认 ANSI，写中文文件会乱码。建议确保脚本文件本身使用 UTF-8 with BOM 或显式 `chcp 65001`。

## 7. 产品 / 用户体验

### P1-30 自编故事仅靠 LLM 解析，失败无重试入口

- 故事 status=error 后只能删除重导。
- **建议**：增加"重新解析"按钮 + 后台保留最近一次 LLM 原始返回供调试。

### P1-31 状态面板信息密度仍偏低

- 当前面板展示 HP/疲劳/金币/位置/章节/物品，但缺少：最近触发事件时间线、记忆摘要、好感度可视化。
- **建议**：右侧分 Tab：状态 / 事件 / 记忆 / NPC。

### P2-32 多设备 / 移动端

- 已有 `mobileSidebar`、`mobileStatus` 简易适配，但未做 PWA、离线缓存、桌面通知权限引导。
- **建议**：渐进式 PWA + Service Worker 缓存最近一次会话。

### P2-33 缺少多语言

- 全站中文硬编码；外文社区无法使用。
- **建议**：抽离 `i18n.ts`，先做 zh-CN / en。

### P3-34 缺少头像 / 个性化主题

- 仅一种科技风浅色主题，长时间使用易疲劳。
- **建议**：暗色主题 + 至少 2 套预设字体（衬线小说体 / 等宽极客体）。

## 10. 一句话总结

> Inkless 已经是一个**功能闭环、模块清晰、叙事质量在线**的小型 AI 互动叙事平台；
> 接下来 1–2 个迭代的关键不是"加更多功能"，而是把 **安全（密钥/默认账号）、数据（迁移/PG）、可观测（结构化日志/trace）、可测（pytest 覆盖）** 这四件事系统化补齐——
> 这能让 v1.x 真正具备"对外开放、长期维护"的能力，并为 v2 引入向量 RAG / 多模态资产 / 协作叙事打下地基。
