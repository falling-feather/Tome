# 部署与运维指南

## 1. 环境要求

### 最低配置
- CPU: 1 核
- 内存: 1 GB
- 磁盘: 5 GB
- OS: Ubuntu 20.04+ / CentOS 7+ / Windows Server 2019+

### 推荐配置
- CPU: 2 核
- 内存: 4 GB
- 磁盘: 20 GB SSD

### 软件依赖
- Python 3.11+
- Node.js 18+
- npm 或 yarn

## 2. 部署方式

### 方式一：脚本部署（推荐）

**Windows:**
```powershell
# 进入项目目录
cd writegame

# 一键部署
.\scripts\deploy.ps1

# 编辑环境变量
notepad .env

# 启动服务
.\scripts\start.ps1
```

**Linux:**
```bash
cd writegame
chmod +x scripts/*.sh
./scripts/deploy.sh
nano .env
./scripts/start.sh
```

### 方式二：Docker 部署

```bash
# 编辑环境变量
cp .env.example .env
vim .env

# 构建并启动
docker compose up -d

# 查看日志
docker compose logs -f
```

### 方式三：手动部署

参见 README.md 中的手动部署步骤。

## 3. Nginx 反向代理（可选）

```nginx
server {
    listen 80;
    server_name your-domain.com;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # SSE 支持
        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 300s;
    }
}
```

## 4. 数据库配置

### SQLite（默认）
无需额外配置，数据文件为 `writegame.db`。

### PostgreSQL（生产推荐）
```bash
# 安装 asyncpg 驱动
pip install asyncpg

# 修改 .env
DATABASE_URL=postgresql+asyncpg://user:password@localhost:5432/writegame
```

PostgreSQL 对大文本存储和并发性能更优，建议生产环境使用。

## 5. 安全注意事项

1. **SECRET_KEY**: 生产环境必须使用随机强密钥
   ```python
   import secrets
   print(secrets.token_hex(32))
   ```

2. **API Key 安全**: API Key 存储在 `.env` 文件和数据库中，确保文件权限正确
   ```bash
   chmod 600 .env
   ```

3. **CORS**: 生产环境应限制跨域来源，修改 `backend/app/main.py` 中的 `allow_origins`

4. **HTTPS**: 生产环境强烈建议配置 SSL

## 6. 备份策略

### SQLite
```bash
# 备份
cp writegame.db writegame.db.bak.$(date +%Y%m%d)

# 恢复
cp writegame.db.bak.YYYYMMDD writegame.db
```

### PostgreSQL
```bash
# 备份
pg_dump writegame > backup_$(date +%Y%m%d).sql

# 恢复
psql writegame < backup_YYYYMMDD.sql
```

## 9. 故障排查

> 如果你是“阿里云 Windows 10 图形界面服务器 + 二级域名 + 自定义端口”这一类环境，建议直接参考更具体的实操文档：
> [docs/07-阿里云Windows10域名部署指南.md](docs/07-阿里云Windows10域名部署指南.md)


| 问题 | 排查方向 |
|------|----------|
| 启动失败 | 检查 Python 版本和依赖安装 |
| 数据库错误 | 检查 DATABASE_URL 配置 |
| LLM 无响应 | 检查 API Key 和网络连接 |
| 前端空白 | 确认 `frontend/dist/` 已构建 |
| 401 错误 | Token 过期，重新登录 |


## 8. GitHub Pages 部署（前端演示站）

### 8.1 工作流
仓库 `.github/workflows/deploy-pages.yml` 在每次推送到 main 时构建 `frontend/` 并部署到 GitHub Pages，输出 SPA 站点（带 404.html fallback、`.nojekyll`）。

### 8.2 一次性配置
1. 仓库 `Settings → Pages → Source` 选 **GitHub Actions**。
2. 仓库 `Settings → Secrets and variables → Actions → Variables` 新增 `VITE_API_BASE`，值为后端可公开访问的 URL（例如 `https://api.your-domain.com/api`）。未设置时默认 `/api`，仅供同源调试。
3. 推送到 main 后在 `Actions` 页查看 `Deploy Frontend to GitHub Pages` 任务，build / deploy 全绿即成功。

### 8.3 常见排查
| 现象 | 排查方向 |
|------|---------|
| 访问 `https://USER.github.io/REPO/` 返回主站 404 | 检查 deploy job 输出的 `page_url` 是否与预期一致；是否与 user-site (`USER.github.io`) 自定义域 / CNAME 冲突；新部署 CDN 通常需 5-10 分钟传播 |
| 资源 404、页面空白 | 确认 `vite.config.ts` 的 `base` 与 `VITE_BASE` 环境变量一致（项目站为 `/REPO/`） |
| API 请求失败 / CORS | 检查仓库 Variables 中 `VITE_API_BASE` 是否指向已开放跨域的后端 |
| 推送后页面未更新 | GitHub Pages CDN 通常需要 5-10 分钟传播；可在 Actions 重新触发 workflow_dispatch |

### 8.4 前端本地构建（与 Pages 同行为）
```bash
cd frontend
npm install
npm run build:pages   # 等价于 cross-env VITE_BASE=/Tome/ vite build
```
