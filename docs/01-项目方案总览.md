# AI互动角色扮演文字游戏项目方案总览

## 1. 项目定位
本项目面向 Web 端，构建一套由大语言模型驱动的互动角色扮演文字冒险系统。目标是同时满足两点：

1. 高自由度对话体验
2. 强设定一致性与剧情可控性

核心方法是将叙事表达与规则决策解耦：

- 后端规则系统负责事实与状态
- LLM 负责语言表达与临场演绎

## 2. 设计原则

1. 显式状态优先：关键数值与状态全部后端持久化，不依赖模型记忆
2. 宏观可控、微观自由：宏观剧情由节点与事件池控制，微观对话由模型生成
3. 动静分离：静态设定固定高权重，动态上下文按预算装配
4. 可恢复与可观测：任何失败路径必须可降级、可回滚、可追踪
5. 长线记忆工程化：使用五级压缩防止上下文溢出与角色漂移

## 3. 范围定义

### 3.1 In Scope

- 基于会话的文字冒险主循环
- 显式状态机与数值系统
- Roguelike 事件池与权重触发
- RAG 世界书检索
- 五级上下文管理与压缩
- 多智能体协作与审计
- 断路器、重试、降级恢复

### 3.2 Out of Scope（首版）

- 实时 3D 场景渲染
- UGC 创作平台
- 多人联机同场剧情同步
- 商业化支付系统（可作为后续阶段）

## 4. 系统总体架构

- 前端层：React 或 Vue（对话展示、操作输入、状态面板）
- API 层：FastAPI（鉴权、限流、会话接口）
- 游戏编排层：状态机、事件池、多智能体编排
- AI 层：LLM 网关、Prompt 管理、输出审计
- 知识层：RAG 检索服务与世界书索引
- 存储层：PostgreSQL + Redis + 向量库
- 观测层：日志、指标、追踪、告警

## 5. 核心交互管线
每轮交互按固定顺序执行：

1. 状态结算
2. 事件判定
3. 世界书检索
4. 记忆压缩
5. Prompt 组装
6. LLM 请求
7. 输出审计与后处理
8. 状态持久化与指标上报
9. 返回前端

建议统一上下文装配格式：

[系统静态设定] + [显式状态变量] + [RAG 条目] + [压缩历史] + [玩家输入]

## 6. 显式状态系统

### 6.1 核心变量（示例）

- affection_score
- trust_score
- health
- fatigue
- money
- current_location
- time_tick
- chapter_id
- quest_flags

### 6.2 状态机建议

- 使用有限状态机或分层状态机
- 状态迁移必须经过后端校验
- 迁移失败要返回可解释原因
- 每轮保存状态快照，支持读档与回滚

## 7. Roguelike 事件系统

### 7.1 事件模型

- 事件定义：前置条件、触发权重、冷却时间、后果
- 事件类别：环境、人物、剧情、资源、风险
- 事件日志：记录触发原因与候选列表

### 7.2 权重建议

weight = base_weight × state_factor × location_factor × decay_factor × player_style_factor

## 8. RAG 世界书系统

采用分层世界书，避免全量塞入 Prompt：

1. 核心设定层：不可变设定（时代、硬规则）
2. 章节设定层：当前章节相关组织、人物、地点
3. 短期设定层：近期冲突、承诺、伏笔

检索流程：

- 查询改写
- 向量召回 + 关键词过滤
- 重排 TopK
- 注入 Prompt

## 9. Prompt 架构

### 9.1 静态层

- 角色人设
- 世界观硬约束
- 护栏规则
- 输出格式规范

### 9.2 动态层

- 当前状态变量
- 最近关键记忆
- 事件客观描述
- 用户输入

### 9.3 工程实践

- 模板版本化
- 关键字段必填校验
- 每次请求保留 Prompt 快照（脱敏）

## 10. 五级上下文管理

1. Snip：滑窗裁剪低价值历史
2. Microcompact：工具输出外置，仅保留引用
3. Collapse：中段对话折叠为剧情摘要
4. Autocompact：深度压缩非活跃历史
5. Reactive：413 等异常时应急清理与回灌关键节点

恢复要求：

- 无论触发哪级压缩，最终必须回灌最近 5 个关键节点
- 人设与技能规则必须置顶恢复

## 11. 多智能体协作模式
建议采用主编排器 + 专职代理：

- 角色代理：角色口吻与即时反应
- 世界观代理：设定与时代一致性
- 叙事代理：节奏与冲突推进
- 审计代理：规则校验与越界拦截

主编排器只输出通过审计的候选结果。

## 12. 技术栈建议（首选）

- 后端：Python + FastAPI
- 数据：PostgreSQL + Redis
- 向量：Milvus 或 pgvector
- 异步任务：Celery/RQ
- 前端：React 或 Vue
- 运维：Docker Compose 起步，后续 K8s

## 13. 非功能性要求

- P95 响应延迟 <= 8 秒
- 接口成功率 >= 99%
- 剧情一致性 >= 90%
- 断路恢复成功率 >= 95%
- 敏感与违规输出可审计

## 14. 风险与对策

- 成本超支：分级模型降级 + Token 预算
- 角色漂移：静态层置顶 + 审计代理
- 检索漂移：分层世界书 + 重排
- 状态死锁：回退检查点 + 强制事件解锁
- 上下文超限：五级压缩 + Reactive 应急

## 15. 首版交付定义

首版可交付（MVP）需满足：

1. 完整一轮会话主循环可稳定执行
2. 显式状态更新可追踪可回滚
3. 至少 20 个事件可用并支持权重触发
4. 基础世界书检索可用
5. 支持基础压缩与失败恢复
6. 有最小监控指标与错误日志
